Chapter 08 · Pattern 3

Event-driven agents join when something happens.

Use this pattern when agents should not wait for direct instructions. In a refund workflow, one event can wake the quality team, another can wake retention, and routine cases stay lean.

Events and triggersFan-out4 quiz questions
1 Core idea

Agents subscribe to the state changes that matter.

The order team handles the initial request. If it discovers a product defect, it publishes a defect event. If the customer is angry or high-value, an escalation event can wake the retention team.

LangGraph translation

You can model events as values in state, then use conditional edges to route to the agents that should respond.

3 Code shape

Events can be simple state updates.

def publish_events(state: RefundState) -> RefundState:
    events = list(state.get("events", []))
    if state.get("defect_found"):
        events.append("defect_found")
    if state.get("support_contacts", 0) >= 3:
        events.append("customer_escalated")
    return {"events": events}

def route_subscribers(state: RefundState) -> list[str]:
    routes = []
    if "defect_found" in state["events"]:
        routes.append("quality_team")
    if "customer_escalated" in state["events"]:
        routes.append("retention_team")
    return routes
Watch out

Event-driven designs are flexible, but tracing can get harder. Keep event names explicit and log which agent responded to which event.
