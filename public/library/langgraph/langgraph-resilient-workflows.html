Chapter 04 · Worked example

When things fail,
keep the workflow useful.

Real systems don't get a clean run every time. Application programming interfaces (APIs) go down, documents arrive corrupted, and some jobs are too big to finish in time. We'll build a LangGraph workflow for legal-contract processing that treats failure as a normal input — not an exception — and still hands back something a human can act on.

3 failure scenarios 3 decision points 5 quiz questions
1 The problem

Resilience isn't a nice-to-have here

A law firm runs hundreds of contracts a day through an automated pipeline: extract key fields, validate legal clauses, generate a summary. Every minute of downtime delays a deal. But the dependencies are flaky — optical character recognition (OCR) services blip, the external legal-clause database goes offline, and some contracts are too long to process whole within the time budget.

What "resilient" means here

The system keeps producing something useful when a piece breaks — a partial extraction, a fallback source, or a clean hand-off to a human — instead of throwing an exception and stopping.

What it isn't

Resilience isn't "never fails." It's having a planned, recorded response for each way the system can fail, so a failure degrades the result instead of destroying it.

Why a graph helps

Each failure mode lives in its own node with its own conditional edge. You can reason about "what happens if OCR is down" without re-reading the whole pipeline.

2 State

State has two jobs, not one

Most tutorials only model the business output. A resilient workflow's state also has to carry how the result was produced — what failed, what filled in for it, and who needs to see that.

class LegalDocumentState(TypedDict):
    # --- business data ---
    document_id: str
    document_content: str
    status: NotRequired[Literal["processing", "completed", "partial", "manual_review"]]
    extracted_fields: NotRequired[dict]
    validation_results: NotRequired[dict]
    summary: NotRequired[str]

    # --- operating context ---
    urgency: Literal["standard", "rush"]
    system_health: Literal["healthy", "degraded", "offline"]
    processing_history: dict   # prior_failures for this doc type

    # --- resilience tracking ---
    retry_counts: NotRequired[dict[str, int]]
    fallback_usage: NotRequired[list[str]]
    error_logs: NotRequired[list[dict]]

    # --- compliance / observability ---
    audit_trail: NotRequired[list[dict]]
    data_sources_used: NotRequired[list[str]]
    compliance_notes: NotRequired[list[str]]

Business fields

What the user actually wants: the extracted fields, the validation results, the summary. This is the part most state schemas stop at.

Resilience fields

retry_counts, fallback_usage, and error_logs exist purely so the workflow — and a human later — can answer "why does this result look the way it does?"

The pattern, generalized

Any resilient agent state needs the same three layers:

  • What happened — the business output.
  • What almost didn't happen — errors, retries, fallbacks.
  • Who needs to know — audit trail, compliance notes.

Skip the last two and your system can still recover from failures — it cannot explain how, which makes debugging and trust-building much harder later.

3 Error handling strategies

Three failure modes, three plans

"Handle errors" is too vague to build from. A resilient node needs a concrete answer to three questions: how do you detect the failure, what's the fallback, and what gets recorded about it.

External API failure

Detect

OCR service or legal database reports degraded/offline, or the call itself errors out.

Fallback

OCR down → use embedded document text. Legal database down → validate against an internal clause library instead.

Record

Increment retry_counts, log the error, append the fallback name, continue with status: partial.

Data quality issue

Detect

Empty content, or a corrupted/unreadable scan that extraction can't make sense of.

Fallback

None, by design — don't let a model guess at legal conclusions from broken input. Route straight to manual_review.

Record

Set status: manual_review, log the failed step, add a compliance note explaining why automation stopped.

Processing timeout

Detect

Document is long or flagged complex (multi-party, master service agreement language) before you've even started extracting.

Fallback

Switch to chunked processing — work the document in pieces and return a partial-but-useful extraction instead of timing out on the whole thing.

Record

Log the timeout risk, mark chunked_processing as a fallback, note the result may be partial.

The shape repeats

Notice all three follow the same three-part template, with different answers each time:

  • Detect — what tells you this failure is happening right now.
  • Fallback — the alternative path that still produces a usable result.
  • Record — what gets written to state so the failure is visible later.

Once you have that template, "handling a new failure mode" means filling in three blanks, not redesigning the graph.

# the LangGraph wiring: state, nodes, and conditional edges
from langgraph.graph import END, START, StateGraph

workflow = StateGraph(LegalDocumentState)
workflow.add_node("assess_document", assess_document)
workflow.add_node("fast_track_extract", fast_track_extract)
workflow.add_node("standard_extract", standard_extract)
workflow.add_node("chunked_processing", chunked_processing)
workflow.add_node("validate_clauses", validate_clauses)
workflow.add_node("generate_summary", generate_summary)
workflow.add_node("manual_review", manual_review)

workflow.add_edge(START, "assess_document")
workflow.add_conditional_edges(
    "assess_document",
    choose_initial_path,
    {
        "manual_review": "manual_review",
        "fast_track": "fast_track_extract",
        "standard": "standard_extract",
        "chunked": "chunked_processing",
    },
)
workflow.add_conditional_edges(
    "standard_extract",
    choose_after_extraction,
    {"manual_review": "manual_review", "validate": "validate_clauses", "chunked": "chunked_processing"},
)
workflow.add_edge("fast_track_extract", "validate_clauses")
workflow.add_edge("chunked_processing", "validate_clauses")
workflow.add_conditional_edges(
    "validate_clauses",
    choose_completion_path,
    {"summarize": "generate_summary", "manual_review": "manual_review"},
)
workflow.add_edge("generate_summary", END)
workflow.add_edge("manual_review", END)

app = workflow.compile()
5 Fallback mechanisms

A fallback is a planned downgrade, not a hope

"Fallback" gets used loosely. In a resilient workflow it means something specific: a pre-built, lower-cost alternative path that's wired into the graph before anything breaks — not error-handling code bolted on after.

In this workflow

  • Embedded text extraction stands in for OCR when the OCR service is down.
  • Internal clause library stands in for the external legal database.
  • Chunked processing stands in for full-document extraction when the document is too big to finish in time.

Each is a real, working alternative — narrower than the primary path, but still useful.

In a typical large language model (LLM) agent

The same pattern shows up as:

  • A smaller/cheaper model as a backup when the primary model errors or rate-limits.
  • Retry with exponential backoff for transient network errors.
  • A cached or templated response when a live generation call can't complete in time.
# the general shape of a fallback, independent of domain
def call_with_fallback(primary, fallback, *args):
    try:
        return retry_with_backoff(primary, *args, max_attempts=3)
    except ServiceUnavailable:
        result = fallback(*args)
        log_fallback_used(name=fallback.__name__)
        return result
Fallbacks have a ceiling

A fallback buys you continuity, not parity. The internal clause library is less thorough than the live legal database — that's why validate_clauses still raises requires_attorney_review more often when it had to use the fallback. A good fallback admits it's a downgrade in the state, instead of pretending the result is as strong as the primary path.

6 Observability for improvement

If you can't see the failure, you can't fix it

Resilience and observability are two sides of the same coin. Handling a failure gracefully in the moment is only half the job — the workflow also needs to leave a trail so someone can later ask "how often does this happen, and where?"

processing_steps
Which node ran, when, success / fallback / failure — replay the whole run after the fact.
error_logs
Node, error type, message, retry count, whether a fallback covered it — the raw material for a "what's breaking" dashboard.
audit_trail
Event, actor, timestamp, data sources used — the compliance answer to "what did the system rely on for this decision?"
From log line to dashboard

None of this is useful trapped inside one run's state. The reason to write error_logs and fallback_usage as structured fields — not free-text print statements — is so they can be streamed out and aggregated downstream into questions like:

  • Which node fails most often?
  • Which fallback gets used most often?
  • Does a particular document type keep landing in manual_review?

That aggregate view is what tells you where to actually invest engineering effort next, instead of guessing.

7 In action

Four contracts, four outcomes

Same graph, four different conditions going in — watch how the outcome tracks the failure, not only the contract content.

DocumentConditionsPath takenOutcome
CON-001 — standard non-disclosure agreement (NDA), parties / dates / clauses all present rush, system healthy, 0 prior failures fast_track_extract → validate_clauses completed
CON-002 — service agreement, missing a liability clause standard, system offline, 1 prior failure standard_extract (OCR fallback) → validate_clauses (database fallback) partial · 2 fallbacks used

CON-004 — "CORRUPTED UNREADABLE SCAN" rush, system healthy, 0 prior failures manual_review (immediately, no extraction attempted) manual_review
The interesting one

CON-003 isn't broken input — every field is technically readable. It lands in manual_review because complexity + a degraded system + a history of failures combine to force chunked (partial) processing, and the partial extraction then trips the high-risk threshold during validation. Three separate "minor" pressures compounded into an escalation — exactly the kind of failure a purely linear pipeline wouldn't have caught.
