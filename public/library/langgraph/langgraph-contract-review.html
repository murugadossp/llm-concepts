What LangGraph actually is

LangGraph models an AI workflow as a graph — and a graph here is built from exactly three things. Get these three, and everything else (persistence, debugging, guardrails) is just detail layered on top.

1 · Nodes

Stateful functions that do a task — analyse symptoms, review a clause. They read the shared state and write updates back. Not stateless: a node both acts and remembers.

2 · Edges

The flow between nodes. Conditional edges route dynamically on the current state — e.g. send cardiac symptoms to cardiology, respiratory ones to pulmonology.

3 · State

Shared memory every node can read and write. It persists through the whole workflow, carrying context so each node can make informed decisions. This is the heart of the design.

Stateless vs. stateful nodes

The single biggest mental shift. Toggle to compare how data moves.

Grows organically

Adding a step means defining one node and wiring its edges — no rewriting the rest. Shared state integrates it automatically, so workflows expand without large code changes.

Intelligible by design

Because state is centralized and every step is tracked, you can visualize the graph, step through execution, and inspect state at each node (LangGraph Studio) — making errors easy to trace.

Seen in the wild

Healthcare diagnostics — an admission node seeds patient state; a diagnostics node updates it; conditional edges route to cardiology or pulmonology based on findings; billing reads the same shared state. Klarna's customer-service assistant is a real production example of this graph-based, stateful approach handling complex, branching conversations at scale.

1 Graph Architecture

Nodes, edges, and conditional logic

In LangGraph a node is a step (a function that reads state and returns an update). An edge is the arrow that moves you to the next node. A conditional edge is a fork: a small router function looks at the state and decides which node runs next. Click any node below to inspect it.

The contract review graph

Every node here is one of two things: a worker that does something (LLM Review), or a router that only decides where to go next (Human Review fans out three ways).

Select a node on the left to see what it does, the state it writes, and the edges leaving it.

Nodes

Plain functions: def node(state) -> dict. They receive the current state and return only the keys they want to change.

Edges

Fixed wiring. add_edge("entry","llm_review") means after entry, always go to review.

Conditional edges

A router function returns the name of the next node. That's how "has queries?" sends you back to the LLM.

# Wiring the contract-review graph
from langgraph.graph import StateGraph, START, END

g = StateGraph(ContractState)
g.add_node("entry", ingest_contract)
g.add_node("llm_review", llm_review)
g.add_node("human_review", human_review)
g.add_node("legal", escalate_to_legal)

g.add_edge(START, "entry")
g.add_edge("entry", "llm_review")
g.add_edge("llm_review", "human_review")

# conditional edge: router reads state, returns next node name
def route_after_human(state):
    if state["decision"] == "approved":   return "done"
    if state["decision"] == "escalate":   return "legal"
    return "llm_review"   # has more queries → loop back

g.add_conditional_edges("human_review", route_after_human, {
    "done": END, "legal": "legal", "llm_review": "llm_review",
})
app = g.compile(checkpointer=checkpointer)
Why you got this right

In the dialogue you spotted that looping back to Contract Entry would re-ingest from scratch — so feedback should flow to a new Human Review node instead, and only loop to the LLM when there are genuine follow-up queries. That's exactly the conditional-edge pattern above.

2 State & Persistence

State schema, reducers, and checkpointing

State is the shared dictionary every node reads and writes. You declare its shape once as a schema. Persisting that state after each node — checkpointing — is what lets a workflow survive a crash, a closed browser, or a week of waiting on a human.

The state schema

Define exactly what flows through the graph. A reducer (like add) says how to merge updates — e.g. append to the feedback history instead of overwriting it.

from typing import Annotated, Literal
from operator import add

class ContractState(TypedDict):
    contract_text: str
    status: Literal["open","llm_reviewed",
        "awaiting_human","escalated","done"]
    # reducer → each node appends, never clobbers
    feedback: Annotated[list, add]
    iteration: int
    decision: str | None

Checkpointing = durable memory

You were right that in-memory state vanishes on restart. LangGraph's answer is a checkpointer that writes the full state after every node. Swap the backend without touching node code:

# dev: in-memory (lost on restart)
from langgraph.checkpoint.memory import MemorySaver
cp = MemorySaver()

# prod: durable on disk / DB
from langgraph.checkpoint.sqlite import SqliteSaver
cp = SqliteSaver.from_conn_string("contracts.db")
# (Postgres saver also available)

app = graph.compile(checkpointer=cp)
app.invoke(inputs, config={"configurable":
    {"thread_id": "contract-8842"}})

Status as a state machine

Your idea — status changing as the contract moves — is exactly how teams make a workflow auditable. Each node advances the status, and because every transition is checkpointed, the sequence is the history.

1
Contract entered
status = "open"
2
LLM finishes review
status = "llm_reviewed" → "awaiting_human" · feedback += [...]
3
Human has queries → loop
iteration += 1 · back to "llm_reviewed"
4
Resolved
status = "done" or "escalated"
Level up — your flagged area

Two things to explore next: state schemas with reducers (so concurrent updates and history merge predictably), and checkpoint threads — each thread_id is one contract's full timeline, and you can rewind to any past checkpoint to replay or branch. That's "time-travel" debugging, and it's the iteration/history mechanism you were reaching for.

3 Transparency & Debuggability

Make every decision auditable

For a legal team, "why did the system do that?" must always be answerable. LangGraph gives you three complementary views — pick a tab.

Your instinct — log the state at the end of each node and show the order — is the foundation of auditability. Stream each step as it happens:

# stream every node execution as it runs
for step in app.stream(inputs, config, stream_mode="updates"):
    for node, update in step.items():
        log.info("node=%s  status=%s", node, update.get("status"))
# → node=entry        status=open
# → node=llm_review   status=awaiting_human
# → node=human_review status=open      (looped: has queries)
# → node=llm_review   status=awaiting_human
# → node=human_review status=done

Different stream_mode values give different granularity — choose what your audit needs:

  • "updates" — what each node changed (compact audit log).
  • "values" — the full state snapshot after each step.
  • "messages" — token-by-token LLM output, for live UIs.
  • "debug" — node entry/exit, timing, and checkpoint writes.

Pair this with LangSmith tracing to get a clickable, visual run timeline — the step-by-step visualization you described, without building it yourself.

Because each step is checkpointed, you can list a contract's entire timeline and inspect the state at any point:

# walk the full history for one contract thread
for ckpt in app.get_state_history(config):
    print(ckpt.metadata["step"], ckpt.values["status"],
          "iter=", ckpt.values["iteration"])
# → 4 done       iter=1
# → 3 awaiting…  iter=1
# → 2 open       iter=0   ← you can resume/replay from here

This is the audit trail and the debugger in one: replay a contract exactly as it happened, or branch from a past state to test a fix.

4 Workflow Control

Guardrails: loops, limits, and humans

Loops are powerful and dangerous. A human who keeps asking questions — or a buggy router — can cycle forever. You named the two essential guardrails: a hard iteration cap, and notifications so a waiting workflow never gets abandoned.

Cap the loop

Drag the cap and watch the simulator's iteration meter. Two layers protect you: a graph-wide recursion_limit, and your own counter in state.

3 
def route_after_human(state):
    if state["iteration"] >= MAX_ITERS:
        return "escalate"   # bail out safely
    ...
# belt & braces: hard stop at the engine level
app.invoke(inputs, config={"recursion_limit": 25})

Pause for a human, don't abandon

Use interrupt to pause the graph and persist its state. The workflow simply waits — for a minute or a month — until a human resumes the same thread. Pair it with a timer that fires your email / SMS reminder.

from langgraph.types import interrupt

def human_review(state):
    # graph pauses + checkpoints here
    verdict = interrupt({"feedback": state["feedback"]})
    return {"decision": verdict}

# scheduled job: nudge stale reviews
if waiting_hours(thread) > 24:
    notify(reviewer, "Contract 8842 needs you")
The failure you're preventing

Without a cap, a "human keeps having queries" loop (or a router bug) runs until it crashes or burns your LLM budget. Without notifications, an interrupt that nobody answers leaves a contract stuck in awaiting_human forever. Both guardrails are cheap; skipping them is how real workflows die quietly.
