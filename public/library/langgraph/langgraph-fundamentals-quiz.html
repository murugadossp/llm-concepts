1 State

The state schema

The state schema defines the shape of the data that flows through every step of the workflow — the one thing all nodes share and can see.

Why it matters

Without a shared schema, every node would need its own ad-hoc way of passing data to the next. With one, any node can read what came before and write a partial update — LangGraph merges it in automatically.

True or false

"In LangGraph, the state schema defines the data structure that flows through each step of the workflow." — True. That's exactly its job.

2 Building blocks

Nodes & edges

A node is a function that takes the current state and returns an updated state. Edges wire nodes together into a flow.

Creating & connecting nodes

  • Create a node as a function: read state in, return a partial update out.
  • Register it on the graph builder so LangGraph knows it exists.
  • Connect nodes with edges that define which node runs next.
  • Branch with conditional edges when the next step depends on state.

What a node is not

Not a visual element, not a database, not the routing logic itself. A node is purely a function: state in, updated state out. The graph structure (edges) is what decides where its output goes.

3 Dynamic behavior

Conditional edges & routing

Conditional edges evaluate the current state and decide which node runs next — that's what makes a workflow adaptive instead of a fixed pipeline.

Worked example

A customer-inquiry router classifies an incoming message, then a conditional edge checks the confidence score: high confidence → auto-respond; low confidence → routed to human review rather than guessed at or discarded.

4 Operability

Visibility & debugging

Because every node reads and writes the same shared state, you can see exactly what happened, in what order, and why.

Clear execution paths

See exactly which nodes ran and in what order.

State inspection

Examine the state at each step to see how the data changed.

Decision transparency

Trace why a conditional edge routed where it did.

Bottom line

This transparency is what makes it possible to pinpoint the exact step and condition behind a bug, instead of guessing — which is what makes the system reliable enough for production.

5 Architecture

Node-based design & scalability

Treating each step as a self-contained node is what lets workflows grow without becoming unmanageable.

Modularity

Each node encapsulates one function — add, remove, or update parts without rebuilding the whole graph.

Composability

Simple nodes combine into complex workflows by changing how they're connected, not how they're written.

Parallel development

Different teams can own different nodes independently, since each one only depends on the shared state contract.

Incremental growth

Start with a simple graph and add nodes over time as requirements grow — no rewrite required.

★ Worked example

Designing a classifier, one decision at a time

Same five ideas above, but built up from nothing — the way you'd actually start a new workflow. Each step adds one piece to the state schema.

{ query: "" } → { query, response } → { query, category, response }
Prompt

How would you define a state schema for an agent that receives a query, processes it, and generates a response?

Your answer

Your answer

Your answer

"Depends on the steps — each step or node in between adds its own state." Exactly right: the schema grows as the workflow grows, it isn't fixed up front.

Your answer

Final schema: { query, category, response } — adding category lets later nodes route and tailor the response instead of treating every query the same.

Node responsibility

"The classification node reads the query from state, classifies it, and updates the category." One job, one state field owned — the same pattern as classify_ticket in the Chapter 03 ticket router.

Conditional routing

"If the category is customer service, route to the customer service node; if product inquiry, follow the product inquiry node." A conditional edge reading one state field to pick the next node — routing logic, not a new node.

Why this matters at scale

Your own takeaway: clear node boundaries let you see where traffic is heavy and which node is misbehaving, so you can debug or scale that one piece without touching the rest. That's the maintainability and scalability argument from sections 4 and 5, arrived at independently.
