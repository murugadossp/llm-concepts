Chapter 06 · Pattern 1

The coordinator keeps the whole workflow in view.

Use this pattern when the work has a clear order. In our refund workflow, one coordinator decides whether the order agent, policy agent, quality agent, or response agent should act next.

Central routingConditional edges4 quiz questions
1 Why it exists

Dependencies are easier when one node owns the sequence.

The coordinator pattern is useful when every step depends on the previous step. You should not check refund policy before you know the order status, and you should not write the final response before quality risk is settled.

Trade-off

The coordinator reduces confusion, but it can become too important. If every tiny decision passes through one node, the graph may become slower and harder to change.

2 Click-through flow

Watch the coordinator route a refund case.

The coordinator is the only agent that talks to every worker. Worker agents update the shared state, then control returns to the coordinator. They do not call each other directly.

All arrows conceptually go through the coordinator: coordinator → worker → coordinator → next worker. This is the key difference from a simple pipeline.

1. Start

The customer says: "My blender arrived cracked. I want a refund."

{ customer_message: "arrived cracked", stage: "initial" }

2. Coordinator → Order agent → Coordinator

The coordinator routes to the order agent because no order facts exist yet. The order agent writes facts, then returns control.

{ stage: "order_checked", order_status: "delivered yesterday" }

3. Coordinator → Policy agent → Coordinator

The coordinator sees order facts are present, then asks the policy agent to check refund eligibility.

{ stage: "policy_checked", policy_result: "eligible" }

4. Coordinator → Quality agent → Coordinator

Because the message mentions damage, the coordinator adds a quality review before final response.

{ stage: "quality_checked", defect_found: true }

5. Coordinator → Response agent

The coordinator sends the completed case file to the response agent, which writes the customer-safe reply.

{ stage: "done", final_response: "refund approved" }

3 Code shape

The coordinator is mostly routing logic.

def coordinator(state: RefundState) -> str:
    if state["order_status"] is None:
        return "order_agent"
    if state["policy_result"] is None:
        return "policy_agent"
    if state["defect_claimed"] and state["defect_found"] is None:
        return "quality_agent"
    return "response_agent"

The graph loops through the coordinator after each worker. That gives one place to manage dependencies and stop conditions.
