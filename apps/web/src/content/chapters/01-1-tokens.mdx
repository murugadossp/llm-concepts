By the end of this lesson, you should be able to look at a sentence and predict why a model may split it into surprising pieces.

## Quick Summary

- A token is a chunk of text: a word, part of a word, punctuation mark, number, space, or symbol.
- The tokenizer turns raw text into token IDs before the model sees it.
- Different models can split the same text differently.

## Tokens: The Pieces The Model Can See

When you type a sentence, a large language model does not receive it as a neat row of human words. First, a tokenizer cuts the text into tokens. A token can be a whole word, a word fragment, punctuation, a number, or even a leading space.

For example, the word `happily` might be one token in one tokenizer and two tokens in another: `happi` + `ly`. This is not a bug. Tokenizers reuse common chunks so the model can handle huge vocabularies without memorizing every possible word.

<ELI5Card title="A token is a puzzle piece">
  If a sentence is a picture, tokens are the puzzle pieces the model is allowed to pick up. Some pieces are large, like a common word. Some are large, like "the". Some are small, like a single letter or punctuation.
</ELI5Card>

## Vocabulary Size: The Model's Dictionary

A model's vocabulary size is the number of unique tokens it has learned to recognize. Modern models typically have vocabularies between **32,000 and 250,000 tokens**. 
* **Large Vocabularies**: Compress text into fewer tokens, which makes processing faster and cheaper, but requires the model to hold a larger embedding matrix (more parameters).
* **Small Vocabularies**: Require less memory to store, but text gets split into many small fragments, increasing the token count and slowing down generation.

## The Multilingual Tax: Why Non-English Costs More

Most tokenizers are trained predominantly on English text. As a result, they recognize common English words as single tokens, but struggle with other languages. 
For example, the word "prevention" is 1 token in English. The Spanish translation "prevención" might split into 3 tokens (`prev`, `enci`, `ón`). 
In languages like Hindi or Japanese, a single word can split into 4–6 tokens. This means non-English users pay a **"multilingual tax"**—using up their context window much faster and paying more money for the same conceptual inputs.

## Token IDs: Text Becomes Numbers

After tokenization, each token maps to an integer identifier from the model's vocabulary. The model does not reason over the letters `q`, `u`, `e`, `e`, `n` directly. It sees an ID that points to a learned entry.

That sounds cold, but it is useful. Once text becomes IDs, the model can look up a learned vector for each token. That vector is the token's embedding.

<ELI5Card title="The badge number is not the meaning">
  Tess's token ID is like a library card number. It helps the model find the right shelf, but the rich information lives in the embedding the model looks up next.
</ELI5Card>

<DeepDive title="Byte Pair Encoding in one minute">
  Byte Pair Encoding starts with individual character bytes (like `l`, `o`, `w`, `e`, `r`) and repeatedly merges the most frequent pairs that appear together in its training documents:
  
  1. Suppose `e` and `r` are the most common pair. We merge them: `er`.
  2. Next, `o` and `w` appear together often. We merge them: `ow`.
  3. Then, `l` and `ow` are merged to form `low`.
  4. Finally, `low` and `er` are merged to form `lower`.
  
  This loop runs thousands of times until our vocabulary reaches its target size (e.g. 50,000). The puzzle-piece analogy breaks down because tokenizers are not trying to discover perfect linguistic meaning or dictionary boundaries. They are optimizing a practical compression problem.
</DeepDive>

## Remember

<RememberCard>
  - A token is the unit the model reads.
  - A token ID is a vocabulary lookup number.
  - Token boundaries are model-specific.
  - More text usually means more tokens, more cost, and more context pressure.
</RememberCard>
