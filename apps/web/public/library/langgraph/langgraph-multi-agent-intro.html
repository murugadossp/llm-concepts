Chapter 05 · Series intro

Multi-agent systems,
without the mystery fog.

A multi-agent system is a group of agents that divide work, share state, and coordinate decisions. This series teaches the three patterns you saw in the slides using one common example: customer refund requests.

1 Mental model

One agent is a worker. A graph is the operating model.

When a customer asks for a refund, one agent could try to do everything. But real work usually has dependencies: check the order, read the policy, inspect the defect claim, decide whether to escalate, and write the final response.

Plain English

LangGraph helps you make that work visible. Each agent becomes a node, the shared case file becomes state, and the arrows decide who works next.

2 Why not one agent?

A single agent is fine until the work needs control.

For a tiny task, one agent is often enough. But refund handling is not one tiny task. It mixes order lookup, policy reasoning, defect analysis, escalation judgment, and customer communication. When one agent owns all of that, mistakes become harder to spot and harder to recover from.

1 Reliability

Smaller agents have narrower jobs. If the quality check fails, the graph can retry that step or route to review without rerunning the whole workflow.

2 Observability

You can see which step produced which fact: order status, policy result, defect decision, escalation reason, and final response.

3 Specialization

The policy agent can be strict, the response agent can be empathetic, and the quality agent can focus on evidence instead of tone.

4 Safer handoffs

Each agent writes structured state, so later agents do not need to guess why a decision was made.

Important trade-off

Multi-agent systems are not automatically better. They add coordination cost, latency, and more places to test. Use them when the work has real boundaries, risk, or routing decisions.

The N-squared problem

If every agent can talk directly to every other agent, communication grows fast. The formula for directed handoffs is n * (n - 1). The arrows below show why this gets messy.

The exact formula for directed handoffs is N * (N - 1), which expands to N² - N. For small systems, the minus N matters. As N grows, the N² part dominates, so people call it N-squared growth or quadratic growth.

Why patterns help
  • Coordinator: reduces the graph to worker handoffs through one hub.
  • Specialist teams: hide internal links behind team interfaces.
  • Event-driven designs: let agents subscribe only to the events they care about.
2B Why LangGraph helps

Enterprise workflows need more than a clever prompt.

A production refund system is not only asking a model to answer. It needs to remember facts, call different capabilities, branch when risk appears, and let humans review sensitive decisions.

1 Stateful processing

The graph carries a shared case file from step to step, so each agent builds on known facts instead of starting fresh.

2 Modular agents

Order, policy, quality, and response logic can evolve independently instead of living inside one giant prompt.

3 Dynamic routing

Conditional edges let the workflow change path when a defect, escalation, or policy exception appears.

4 Human feedback loops

The graph can pause for approval before issuing high-value refunds, goodwill credits, or risky customer messages.

Traditional LLM limits
  • Context management: one prompt can lose track of which facts are settled and which are uncertain.
  • Coordination complexity: as agents grow, direct handoffs become hard to trace.
  • Workflow rigidity: fixed chains struggle when different cases need different paths.
  • Integration hurdles: real systems need tools, APIs, databases, and review steps around the model.
3 The map

Three patterns cover most beginner use cases.

They are not enemies. A mature system often combines them. For learning, we separate them so you can see the job each one does.

1 Coordinator

A central coordinator reads the current state and routes the workflow to the next agent. Best when the order matters.

central control

2 Specialist teams

Agents are grouped into focused teams, like order, quality, and retention. Best when work maps to departments.

structured teams

3 Event-driven

Agents wake up when the state records an event they care about. Best when work is reactive and conditional.

responsive
