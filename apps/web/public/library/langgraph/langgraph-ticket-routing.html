Routing support tickets
without a single queue.

A SaaS team gets hundreds of tickets a day, all dumped into one queue. We'll build a LangGraph workflow that classifies each ticket, assesses urgency, sends unclear ones through an extra review step, and routes the rest straight to the right specialist.

4 nodes 1 conditional branch 5 quiz questions
1 The problem

One queue doesn't scale

Billing questions, technical bugs, feature requests, and account issues all land in the same place today. Urgent, production-impacting tickets wait behind routine ones, and nobody is automatically flagging the cases the system isn't sure how to classify.

Classify

Work out the ticket's topic — billing, technical, feature request, or account management.

Assess urgency

Scan for urgency signals — "urgent", "outage", "blocked", "ASAP" — independent of topic.

Route

Combine both signals to pick a destination team, escalating high-urgency technical issues to senior engineers.

2 State

The shared state

Every node reads from and writes to the same TicketState — the original ticket plus whatever each node has figured out so far.

class TicketState(TypedDict):
    ticket_content: str
    category: NotRequired[TicketCategory]              # billing | technical | feature_request | account_management | other
    classification_confidence: NotRequired[float]
    urgency: NotRequired[UrgencyLevel]                  # low | medium | high | urgent
    routing_decision: NotRequired[RouteDecision]
    metadata: NotRequired[dict]                         # e.g. review flags
Why NotRequired

Fields start empty and get filled in as the ticket moves through the graph — category after classification, urgency after the urgency check, and so on. Nothing needs to be known up front except the raw ticket text.

3 Workflow nodes

Four focused nodes

Each node does exactly one job and only touches the state fields it owns.

1 classify_ticket

Reads: ticket_content  ·  Writes: category, classification_confidence

Scans the text for keywords — "invoice/refund" → billing, "bug/crash/API" → technical, "feature/enhancement" → feature_request, "login/password" → account_management. No match → other at low confidence.

2 assess_urgency

Reads: ticket_content  ·  Writes: urgency

Independent keyword scan: "urgent/down/outage/blocked" → urgent, "ASAP/production/cannot work" → high, "issue/problem/soon" → medium, otherwise low.

3 additional_review

Reads: metadata  ·  Writes: metadata

Only runs for uncertain tickets. Stamps the state with needs_additional_review: true and a reason, so a human or a second model pass can pick it up before routing.

4 route_ticket

Reads: category, urgency  ·  Writes: routing_decision

Technical + high/urgent → senior_engineer; technical otherwise → technical_support; billing/feature/account map to their specialist teams; everything else → general_support.

Read this as

Both branches converge back on route_ticket — review only ever adds a step, it never skips routing. There's exactly one decision point in the whole graph.

5 Conditional logic

The code behind the branch

After assess_urgency, a single conditional edge decides whether the ticket is clear enough to route immediately, or needs a review step first.

def choose_review_path(state: TicketState) -> Literal["additional_review", "route_ticket"]:
    confidence = state.get("classification_confidence", 0)
    category = state.get("category", "other")
    if category == "other" or confidence < 0.70:
        return "additional_review"
    return "route_ticket"

5 In action

Four tickets through the graph

Same workflow, four different paths — notice which one trips the review branch.

TicketCategoryConfidenceUrgencyRouted to
"I was charged twice on my latest invoice and need a refund." billing 0.90 low billing_specialist
"Urgent: our production API integration is down and users are blocked." technical 0.88 urgent senior_engineer
"I cannot log in to my account after resetting my password." account_management 0.86 low account_specialist
"Something strange is happening and I need help figuring out the right team." other 0.50 → review low general_support
The one that branched

The last ticket's category came back other at 0.50 confidence — below the 0.70 threshold — so additional_review ran first and stamped metadata.needs_additional_review = true before routing to general support. The other three were confident enough to skip straight to routing.
