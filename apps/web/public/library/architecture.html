01 · Scope §1

Goals & non-goals

Six months to ship eleven chapters, paid tier, multi-tenant teams, and an AI tutor. What we will not ship is just as load-bearing.

In scope v1 goals

  • Ship 11 illustrated chapters as the content product
  • Authenticated accounts with progress, bookmarks, notes
  • Paid Pro tier — gated chapters + AI tutor
  • Multi-tenant organizations with team seats & invoiced billing
  • In-page AI tutor — Claude streaming, chapter-aware, rate-limited by tier
  • Solid learning-app design system — light + dark, system-preference, persisted
  • MDX-driven authoring — drop a file in /content/chapters, lesson appears

Out of scope non-goals

  • Native mobile app — web is mobile-responsive
  • Live model demos with user-supplied API keys (CORS + key pain)
  • Translations / i18n — English only at v1
  • User-generated courses (UGC instructor mode)
  • Built-in live chat / community — Discord covers v1
  • Self-hosted on-prem deploy — cloud-only

05 · Authoring §5

Lesson schema

Drop a .mdx file in src/content/chapters/ with this frontmatter. Validated by Zod at build. Root chapters and nested sub-lessons appear in nav, sidebar, sitemap, and OG cards automatically.

---slug: "01-1-tokens"title: "What Is a Token?"navTitle: "Tokens"chapterNumber: 1lessonNumber: 1parentSlug: "01-foundations"section: "foundations"difficulty: "beginner"estimatedMinutes: 8tier: "free"prereqs: []characters: ["Tess"]ogImageVariant: "foundations"updatedAt: "2026-05-23"authors: ["murugadoss"]summary: >  Tokens are the chunks of text an LLM reads  before anything becomes model input.---
slug — URL segment. Also the lookup key in prereqs and progress tables.
chapterNumber / lessonNumber — Sort order across the curriculum and inside a chapter module.
parentSlug — Optional. When present, this file appears as a nested lesson under its parent chapter in the sidebar.
tier — free renders directly. pro uses explicit <FreePreview> / <ProOnly> boundaries.
characters — Which SVG components to preload. Powers the persona strip.
ogImageVariant — Routes to a preset OG card template, rendered at build by /api/og/[slug].
summary — Used in the chapter hub, sitemap, RSS, and meta description.
Adding a chapter or sub-lesson has zero code changes. Write MDX → commit → Vercel preview deploys it → it appears everywhere. No registry to edit.
06 · Persistence §6

Data model

Drizzle schemas in src/db/schema.ts. Multi-tenant from day one: every user has a personal org, so every row has an orgId path.

Identity Billing Content ADR-008 abstraction
Entitlements are the single source of truth. Code never asks "does this user have a Pro subscription?" — it asks "does this user have the chapters:pro entitlement?". This decoupling powers bundles, grants, trials, and provider swaps with zero access-check changes.
07 · Identity §7

Auth & multi-tenant flow

Sign-up creates a personal org so every user always has an orgId. The multi-tenant code path is uniform from day one.

Why the auto-personal-org pattern? Every user has an orgId from the first request. Server components never need to branch between "personal user" and "team user". When the user later upgrades to Team, the same code path now serves multiple seats.
08 · Money §8

Payments & entitlements flow

Lemon Squeezy is merchant-of-record (handles global tax). Webhooks grant entitlements; access checks never know about the billing provider.

★ ADR-008 in action. Step 7 — the entitlements grant — is the only call any access-check code needs. Subscriptions, manual grants, lifetime deals, promo codes, and a future Stripe migration all write to the same table. Access logic stays untouched.
09 · Intelligence §9

AI tutor flow

Claude streams answers back through SSE, grounded in the current chapter. Entitlements and rate limits act as inline guards before any token leaves the server.

10 · Delivery §12 · ADR-011

Build plan — 3 milestones, engagement-gated

Two committed milestones with an explicit engagement gate between them, plus a demand-driven third. Each phase ships against an exit-criteria checklist. Only Milestone 1 is committed — the rest waits for validation.

M1 Content MVP

Prove the learning experience

Weeks 1–4 · committed

Next.js scaffold, theme system, MDX pipeline, all component primitives, three flagship chapters (Foundations, Agents, MCP) with interactive widgets, hub page, email capture, SEO foundations, PostHog analytics.

Exit criteria
  • 3 chapters render in light + dark on desktop & mobile
  • Lighthouse Perf/A11y/SEO ≥ 95 across pages
  • Sitemap, RSS, OG card per chapter validate
  • Newsletter signup → confirmation → list membership works
  • Preview deploys per PR; main → prod
Engagement gate to M2 — any one triggers
  • ≥500 newsletter signups
  • ≥50% scroll-completion on at least one flagship chapter
  • Clear qualitative demand (replies asking for more / paid)
  • If none in ~6 wks → iterate on content, don't start M2
M2 SaaS v1

Convert validated learning into a sustainable product

Weeks 5–14 · only after gate passes

Five sub-phases run sequentially.

M2.A · Accounts & progress (weeks 5–6)

Clerk auth (email magic link + Google + GitHub), Neon + Drizzle, schemas with internal UUIDs + clerkUserId text columns (ADR-013), Clerk webhook sync, dashboard shell.

  • Signup → read → bookmark → re-login → see (E2E)
  • Personal org auto-created on signup
  • Progress updates on 80% scroll heuristic
M2.B · Remaining 8 chapters (weeks 7–8)

Chapters 2–5 + 8–11 authored. `<FreePreview>` / `<ProOnly>` boundaries added (ADR-014) but gating dormant. Pagefind search activates at ≥6 chapters (ADR-010).

  • All 11 chapters published
  • Pagefind index builds clean
M2.C · Payments & gating (weeks 9–10)

Lemon Squeezy, webhook handler with idempotency, `entitlements` table (single source of truth, ADR-008), `<ProOnly>` gating live, `/pricing`. Chapter 7 reclassifies as Pro with 7-day grace email.

  • Test checkout grants entitlement <60s
  • `<ProOnly>` shows paywall to unentitled users (no 25% percentage cut)
  • Cancel at period end works; pay-fail 3-day grace works
  • One $1 production purchase succeeds
M2.D · AI tutor with five cost controls (weeks 11–13)

Streaming /api/tutor with Haiku default + Sonnet escalation + prompt caching + history summarization + credits + $20/user/mo ceiling (ADR-012). TutorChat with credits meter. Rate limit via Upstash.

  • Streaming works on desktop + mobile
  • Credits decrement; renewal resets
  • Cost-ceiling kill switch verified
  • Avg cost per Pro user < $5/mo in PostHog
M2.E · Hardening (week 14)

Sentry coverage, BetterStack uptime, structured logs with PII redaction, Lighthouse CI gates deploys, privacy + ToS + refund policy live, support inbox.

  • All routes covered by Sentry
  • Public status page live
  • Legal pages live
M3 Teams

B2B revenue — only when demand is real

Demand-driven · ADR-011 / P2.8

Gated on inbound demand. Until then, the data abstraction is in place (orgId nullable, entitlements check user OR org), but no team UI, seats, invoicing, or SSO.

Gate to start
  • ≥3 unsolicited team inquiries within a quarter
Scope (only when gate passes)
  • Clerk Organizations exposed in UI
  • /teams pages: members, billing, settings
  • Seat enforcement at invite time
  • Invoiced billing variant in Lemon Squeezy
  • Optional SSO via Clerk enterprise
  • Team credit pools (2,000/seat poolable)
11 · Economics §13

Cost model

Monthly run-rate at four scale points. Excludes founder time. Breakeven sits well below 100 Pro users.

Scale Active users Pro users Tutor msgs/mo Monthly cost
M1 (Content MVP) up to ~500 0 0 ~$0
M2 launch ~100 ~5 ~2,000 ~$35
Mid-scale ~1,000 ~50 ~30,000 ~$300
Healthy product ~10,000 ~500 ~300,000 ~$3,000
Revenue check at 1k / 50 Pro: ~$950/mo gross → ~$700 after LS fees → ~$400 net of infra. Breakeven well below 100 Pro users. Tutor numbers reflect ADR-012 controls (Haiku default at ~$0.005/msg, ~15% Sonnet at ~$0.025/msg). Prior plan's $216/mo blowout case is bounded by the $20/user/mo ceiling.
12 · Decisions §14

Risks & ADRs

Architecture Decision Records — what was decided, why, and how hard it would be to walk back. Click to expand each.

ADR-001Next.js 15 + MDX over Astro

We need a SaaS layer within six months — paid courses, multi-tenant, auth, AI tutor. A single Next codebase keeps content + app cohesive; Astro would force a second stack to ship the app surface.

Reversible? Hard later.Domain: foundation
ADR-002Solid learning-app UI (light + dark) over comic-only aesthetic

Modern aesthetic pairs better with paid-product expectations; theme switching is table-stakes. Opaque surfaces preserve readability and hierarchy better than transparent glass for long-form lessons. Characters are retained and restyled for the theme-aware UI.

Reversible? Easy.Domain: visual
ADR-003Clerk over NextAuth / Supabase Auth

Organizations (multi-tenant) built in, free up to 10k MAU, best Next.js integration. Lock-in is real — mitigated by the lib/auth.ts helper abstraction.

Reversible? Medium.Domain: identity
ADR-004Neon over Supabase / PlanetScale

Postgres + serverless + branchable. Per-PR isolated databases are huge for safety; preview deploys never touch prod data.

Reversible? Medium.Domain: data
ADR-005Drizzle over Prisma

TypeScript-native, edge-friendly, simpler migrations, smaller bundle. Less ecosystem maturity than Prisma but improving fast.

Reversible? Medium.Domain: data
ADR-006Lemon Squeezy over Stripe

Merchant of record handles global tax — saves ~2 weeks of compliance work for a solo founder. Trade ~2% extra fee for that time. Reversible thanks to ADR-008.

Reversible? Medium.Domain: billing
ADR-007Single domain, single codebase over Astro + Next split

Easy lesson authoring across free + paid is the dominant requirement. Two-stack operational overhead is not justified at this scale. Future option: extract (marketing) route group to its own Astro site if SEO demands.

Reversible? Yes, at scale.Domain: foundation
ADR-008★ Entitlements table abstraction over direct subscription checks

Decouples access control from billing provider. Lets us add grants, trials, bundles, lifetime offers, switch billing provider — all without touching access-check code. This is the abstraction that makes ADR-006 reversible.

Reversible? N/A — this is the abstraction.Domain: access control
ADR-009Glass-compatible character restyle over dropping characters

Characters are a memorability asset (BLUEPRINT §4.4). Restyling preserves the asset while modernizing the visual language — cleaner outlines, gradient fills, theme-aware via tokens.

Reversible? Per character.Domain: visual
ADR-010Pagefind over Algolia

Free, static, zero infra. Algolia is overkill until content > 100 pages. Activated only when ≥6 chapters are live (per reviewer P2.4).

Reversible? Easy.Domain: search
ADR-011 ★Two-milestone build with engagement gate

Per reviewer P1.1 — separates "did the content land?" from "did the business work?". The prior plan carried launch, monetization, team sales, AI tutor, and content production all at once. The new structure validates the learning experience first, then commits to SaaS only when engagement metrics (≥500 signups, ≥50% completion, or qualitative demand) trigger the gate. Teams milestone gated separately on ≥3 unsolicited inquiries.

Reversible? Easy.Domain: delivery
ADR-012 ★Tutor cost controls — credits + Haiku default + Sonnet escalation + caching + summarization + ceiling

Per reviewer P1.3 — "almost no one will" was not a cost control. The redesign applies five compounding mitigations: (1) Haiku default not Sonnet (~5× cheaper), (2) Sonnet escalation user-initiated at 4× credits, (3) prompt caching on chapter context (~10% input price), (4) conversation summarization after 6 turns, (5) credits + hard $20/user/mo ceiling. Worst-case heavy Pro user now ~$3.25/mo cost vs prior $216/mo blowout.

Reversible? Easy (tune credit grants).Domain: economics
ADR-013 ★Internal UUIDs + clerkUserId/clerkOrgId text columns

Per reviewer P1.2 — Clerk IDs are user_.../org_... strings, not UUIDs. The original schema declared users.id (uuid, pk) mirrors Clerk userId which would break sync at runtime. New shape: internal random UUIDs as primary keys, plus clerkUserId/clerkOrgId text columns with unique indexes. Decouples internal IDs from any single auth provider (which ADR-003's spirit always required).

Reversible? N/A — this is the right shape.Domain: data model
ADR-014 ★<FreePreview> / <ProOnly> MDX boundary components, not 25% cuts

Per reviewer P2.5 — MDX is component-heavy, so a percentage-of-content cut can split components mid-render, leak deep-dive content, or produce awkward previews. Authors mark the boundary explicitly: <FreePreview> always renders, <ProOnly> checks entitlement and renders children or a fallback. Authors keep full control of the teaser shape.

Reversible? Easy (thin wrappers).Domain: content
ADR-015 ★Target major versions instead of "latest"

Per reviewer P2.7 — the prior spec claimed "version pins" but used latest throughout, which drifts. Major-pinning (e.g. ^15.4, ^19.0) gives reproducibility at scaffold time while allowing patch updates via lockfile. Items not needed until Milestone 2 are flagged (M2) so M1 install surface stays small.

Reversible? N/A — operational.Domain: tooling

Known risks worth watching

  • AI-tutor cost runaway — addressed by ADR-012's five-layer defense (Haiku default, Sonnet escalation, prompt caching, summarization, credits + $20/user/mo ceiling). Backstop kill switch flags accounts > $20/mo for manual review.
  • Lemon Squeezy fee drag at scale argues for Stripe migration. ADR-006 is reversible thanks to ADR-008 (entitlements abstraction).
  • Visual hierarchy regression if transparent effects creep back into the learning shell — keep sidebar, top nav, chapter hero, and article panels on opaque theme surfaces.
  • Clerk lock-in above 10k MAU — pricing escalates. The lib/auth.ts boundary + ADR-013's internal UUIDs make a future swap tedious but not catastrophic.
  • Content MVP gate doesn't trigger — most likely outcome and the gate's whole point. Iterate on content; don't paper over with SaaS scope.
